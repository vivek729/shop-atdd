# 2026-06-24 08:24 UTC — Rename `contract` suite → `provider-verification` (+ drop standalone frontend layer-4 suite)

## TL;DR

**Why:** The component-tier suite `id: contract` is labelled `Consumer Contract (Pact)` on **both** sides, but on the backend it actually runs the **provider** verification (`BackendPactVerificationTest` / `*.backend.contract.*`), so the name is backwards there. Layer 4 in the locked symmetric 4-layer taxonomy is **provider verification**, which exists **only on backends** — the frontend is consumer-only and emits its contract from suites 2+3 (`integration` + `component`) as a union, so it has **no** standalone layer-4 suite. This plan renames the suite by what it does and drops the frontend's mislabeled standalone suite.
**End result:** The 6 backend/monolith `component-tests.yaml` files expose `provider-verification` as their layer-4 suite (`suiteGroups.all: [unit, integration, component, provider-verification]`); the frontend-react `component-tests.yaml` drops its `contract` suite entirely and exposes 3 suites (`suiteGroups.all: [unit, integration, component]`); the component-tier docs follow; the acceptance-stage external-system `contract-*` suites and the `contracts/` `.pact` output folder are left **unchanged**.

## Outcomes

- Backend + monolith (6 files: backend-java, backend-dotnet, backend-typescript, monolith-java, monolith-dotnet, monolith-typescript): suite `id: contract` → `id: provider-verification`; suite `name: Consumer Contract (Pact)` → a provider-verification label; `suiteGroups.all` ends in `provider-verification`.
- Frontend-react: the standalone `contract` suite (`npm run test:pact`) is **removed**; `suiteGroups.all` = `[unit, integration, component]`; no layer-4 suite remains. (Consumer-contract *emission* lives in `integration` + `component` per the Target state — this plan only deletes the suite entry; it does **not** move emission, which is owned by `1801`/`1944`/`1957`.)
- `gh optivem component test run --suite provider-verification` selects the renamed backend suites; `--suite contract` no longer resolves (hard rename, no alias — see Open questions).
- `docs/pipeline/commit-stage.md` references the component-tier suite by its new name; the consumer-vs-provider distinction is corrected.
- **Untouched on purpose:** the `contracts/` repo folder (`.pact` output dir); the acceptance-stage external-system suites `contract-stub` / `contract-stub-isolated` / `contract-real` (and their `mod11-` legacy aliases) defined in `system-test/*/tests.yaml` and pinned by the `*-acceptance-stage*.yml` workflows; `docs/pipeline/acceptance-stage.md`.
