# 2026-07-01 06:00:00 UTC — Gate stateful acceptance-stage steps to main-branch runs only

## TL;DR

**Why:** The acceptance-stage workflows (`*-acceptance-stage.yml` and `*-acceptance-stage-cloud.yml`, 12 files across monolith/multitier × java/dotnet/typescript) can be run against any commit — including a PR branch's commit-sha via `workflow_dispatch` — but unconditionally tag Docker images and publish a git tag as if every run were an official release candidate. Running the stage against a PR branch to validate a build should be safe to do freely; it should never mutate GHCR tags or push git tags.
**End result:** The acceptance stage can be executed from a PR branch to exercise deploy + test suites (smoke/acceptance/contract/e2e) + sonar with zero side effects, while RC image tagging (`Tag Docker Images for Prerelease`) and git tag publishing (`publish-tag` job) execute only when the run is against `main`. All 12 acceptance-stage workflow files behave consistently.

## Outcomes

- Running the acceptance stage against a PR branch (e.g. dispatching with `commit-sha` set to a PR branch's HEAD) runs deployment + all test suites + sonar exactly as today, with no image tags pushed to GHCR and no git tag published.
- Running the acceptance stage against `main` (scheduled cron, or manual dispatch with no `commit-sha` / a main commit-sha) behaves exactly as it does today — RC image tagging and git tag publishing still happen.
- The main-vs-PR distinction is computed once per run and consumed as a single boolean gate, not re-derived ad hoc at each stateful step.
- The gating logic and its `if:` placement are identical across all 12 acceptance-stage files (monolith + multitier, java/dotnet/typescript, latest + cloud variant) — no per-language drift.
- Non-stateful steps (deploy-docker-compose, wait-for-endpoints, all `gh optivem system-test run` suites, sonar analysis, `Compose Prerelease Version` computation) are unaffected and continue to run regardless of branch. Sonar stays ungated — it's already keyed per-branch/PR in SonarCloud and isn't release-tagging state.
- Legacy acceptance-stage variants (`*-acceptance-stage-legacy.yml`) are untouched — they don't do RC tagging or git tag publishing at all, so the gate doesn't apply to them.
- README status badges for commit-stage, acceptance-stage, acceptance-stage-legacy, qa-stage, qa-signoff, and prod-stage (all 6 language/tier blocks) reflect `main`-branch runs only — a manual PR-branch validation dispatch of acceptance-stage never flips these badges, keeping them a trustworthy signal of the actual release pipeline's health.

## Steps

- [ ] Step 1 (decided): Detect "is this a main-branch run" by whether `COMMIT_SHA` (the actual commit under test — `inputs.commit-sha || github.sha`) is an ancestor of `main`, **not** by which branch the workflow was dispatched from. The dispatch-branch picker (`github.ref_name`) only selects which copy of the workflow YAML runs — it's decoupled from `commit-sha` and GitHub's UI defaults the picker to main regardless of intent, so gating on it would silently skip nothing in the common real-world flow (dispatch from main, pass a PR's `commit-sha`). Reuse the existing `optivem/actions/check-sha-on-branch@v1` composite action (already used identically in `monolith-java-commit-stage.yml`'s `check-on-main` step to gate Docker image push) — it takes `commit-sha` (default `github.sha`) and `base-branch` (default `main`), runs `git merge-base --is-ancestor`, and outputs `on-branch=true/false`, failing hard (not swallowing to `false`) on an indeterminate result (bad SHA, missing ref). Add a `Check Commit on Main` step (same name/id pattern as commit-stage: `check-on-main` / `on-branch`) in the acceptance-stage `check` job, passing `commit-sha: ${{ env.COMMIT_SHA }}`, and surface its `on-branch` output as a `check` job output so downstream steps/jobs reuse it via `needs.check.outputs.on-branch` instead of re-deriving it.
- [ ] Step 2: Identify the exact set of "stateful" steps/jobs to gate on `needs.check.outputs.on-branch == 'true'`. Confirmed candidates: `Tag Docker Images for Prerelease` step (pushes tags to GHCR) in the `run` job, and the entire `publish-tag` job (pushes a git tag). Confirm whether `Compose Prerelease Version` should stay ungated (it's a pure computation, not a side effect) — current assumption: yes, stays ungated.
- [ ] Step 3: Implement the gate in `monolith-java-acceptance-stage.yml` as the reference implementation — add the `Check Commit on Main` / `check-on-main` step to the `check` job, expose its `on-branch` output as a `check` job output, and add `if: needs.check.outputs.on-branch == 'true'` to the tagging step and the `publish-tag` job. Mirrors `monolith-java-commit-stage.yml`'s existing use of the same `check-sha-on-branch` action (there it gates `Compose Dev Version`, the dev-version image tag, `Log in to GHCR`, `Build and Push Docker Image`'s `push:` input, and `Compose Digest URL` — all via same-job `steps.check-on-main.outputs.on-branch` since commit-stage keeps everything in one job). The only structural difference: acceptance-stage already splits `check`/`run`/`publish-tag` into separate jobs, so `on-branch` must be promoted to a job-level output on `check` and consumed as `needs.check.outputs.on-branch` in the other jobs, instead of the same-job `steps.*` reference commit-stage uses. Commit-stage also builds the Docker image unconditionally and only gates the push — same shape as our plan running tests unconditionally and only gating the tag/publish steps.
- [ ] Step 4 (decided — YES, needs a distinct message): The `summary` job's `render-system-stage-summary@v1` action requires `success-version` to be non-empty whenever `stage-result == 'success'` (see its `validate.sh`), and `stage-result` will still read as `success` on a non-main run since tests passed — only tagging was skipped. The shared action also has no dedicated field for a "success with caveat" state (only `success-version`/`success-artifact-ids` for success and `skipped-reason` for skipped), and it lives in a separate repo (`optivem/actions`) with its own release/versioning — extending it is possible but out of scope for this shop-repo plan. Recommended fix, entirely within the shop workflow file: keep `needs.run.outputs.version` itself clean/unmodified (it's still used as real data — `publish-tag`'s `tag:` input — even though that job is skipped for non-main), but adjust only the *display* string passed into the summary's `success-version` input: `${{ needs.check.outputs.on-branch == 'true' && needs.run.outputs.version || format('{0} (not published — non-main run)', needs.run.outputs.version) }}`. `success-artifact-ids` already naturally comes out empty on non-main since `tag-images` is skipped, so no artifacts line renders either way.
- [ ] Step 5: Port the identical gate to the remaining 11 acceptance-stage files (monolith-dotnet, monolith-typescript, multitier-java/dotnet/typescript × latest + cloud variant), preserving each file's existing structure.
- [ ] Step 6: Verify with `./compile-all.sh` is not applicable (workflow-only change) — instead validate via `workflow-comparator` agent or manual diff-review across all 12 files to confirm zero drift, plus a real dispatch test (main run and PR-branch run) if feasible.
- [ ] Step 7 (decided — README badges, independent of the gating implementation above): Add `?branch=main` to all badge URLs in `README.md` for `commit-stage` (all `*-commit-stage.yml`), `acceptance-stage.yml`, `acceptance-stage-legacy.yml`, `qa-stage.yml`, `qa-signoff.yml`, `prod-stage.yml`, each ×6 language/tier blocks (monolith-java/dotnet/typescript, multitier-java/dotnet/typescript) — 36 badges total. `commit-stage` badges currently use `?event=push` instead — replace with `?branch=main` so the badge reflects "main branch, any event" (push or workflow_dispatch against main) rather than "push event only," making it consistent with every other badge and no longer coupled to which trigger fired. GitHub's native badge `?branch=` filter reflects the *dispatch-branch picker* (the same `github.ref_name` concept Step 1 deliberately avoided for the RC-tagging gate, since it can diverge from `commit-sha`) — that's an accepted limitation for a display-only badge, not a correctness gate, so it's fine here even though it wouldn't have been safe for Step 1's actual state-mutation gate.
